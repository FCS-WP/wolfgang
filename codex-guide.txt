WOLFGANG ETHOS - CODEX BLOCK BUILD GUIDE

Purpose
This project is the Wolfgang Ethos WordPress child theme built on AI Zippy. Use this guide when another Codex instance needs to build or edit blocks correctly without relying on chat history.

Active project
- Work directly in: /home/rio/fcs/wolfgang
- Parent theme: src/wp-content/themes/ai-zippy
- Child theme: src/wp-content/themes/ai-zippy-child
- Prefer child theme edits. Do not edit parent theme files unless explicitly requested.

Core rule
Build reusable Gutenberg blocks in the child theme. Do not hard-code page sections into templates unless the user asks for a template-only change.

Block categories
- Site-level blocks use category: ai-zippy-site
  - Header, footer, popup, global site UI.
- Page/section blocks use category: ai-zippy-page
  - Home sections, about sections, contact sections, service sections.
- Category registration lives in:
  - src/wp-content/themes/ai-zippy-child/inc/hooks/block_categories.php

Block source and build output
- Source blocks:
  - src/wp-content/themes/ai-zippy-child/src/blocks/<block-slug>/
- Built blocks:
  - src/wp-content/themes/ai-zippy-child/assets/blocks/<block-slug>/
- Do not edit built assets directly. Edit source files and rebuild.

Standard block files
Every dynamic block should normally include:
- block.json
- index.js
- edit.js
- save.js
- render.php
- style.scss
- editor.scss
- view.js only when frontend JavaScript is needed

Dynamic block rules
- save.js returns null.
- render.php outputs frontend markup.
- Escape output in PHP:
  - esc_html for text
  - esc_url for URLs
  - esc_attr for attributes
  - wp_kses_post only for controlled rich text
  - custom wp_kses allowlists for iframe/embed HTML
- Keep class names scoped to the block.
- Make user-facing content editable through attributes, RichText, InspectorControls, MediaUpload, URLInputButton, etc.

Global source:

Theme palette lives in:

All website UI should follow the Wolfgang black/lime/white system unless the user provides a new direction.

Shared section helpers
Use the shared helpers for page section blocks:
- src/wp-content/themes/ai-zippy-child/src/blocks/_shared/section-controls.js

Import pattern:
```js
import {
	SectionControls,
	sectionAttributes,
	getSectionClassName,
	getSectionStyle,
} from "../_shared/section-controls";
```

Required section behavior
Every page section block should support:
- layout: boxed / wide / full
- background color
- text color when useful
- padding top
- padding bottom
- margin top
- margin bottom

Use these global classes:
- .az-section
- .az-section--boxed
- .az-section--wide
- .az-section--full
- .az-section__inner or .az-section-inner

Frontend PHP should mirror the same behavior:
- Add classes: <block-class> az-section az-section--<layout>
- Add CSS variables for background, text color, padding, and margin.
- Wrap content in .az-section__inner.

Responsive rules
- Always design for desktop, tablet, and mobile.
- Use grid/flex layouts that collapse cleanly below about 900px and 767px.
- Use stable dimensions for cards, media, icons, and controls.
- Do not use viewport-scaled font sizes alone. Use clamp() with sane min/max.
- Avoid text overlap and fixed heights that clip content.

Build commands
- Rebuild child blocks after block source changes:
  - npm run build:child-blocks
- Rebuild global SCSS/theme assets after src/scss or Vite asset changes:
  - npm run build
- npm run build also rebuilds parent and child blocks.

Verification checklist
After edits:
- Run php -l on changed PHP render/hook files.
- Run git diff --check on changed files.
- Run npm run build:child-blocks for block edits.
- Run npm run build for global SCSS/theme asset edits.

Known build note
The parent product-showcase Sass may emit deprecation warnings for lighten(). The build can still succeed. Do not edit parent code just to silence this unless requested.

Sandbox note
If direct writes to /home/rio/fcs/wolfgang are blocked by the sandbox, stage files in /tmp and copy them into the project with approved commands. This is a tooling workaround, not a project convention.

Implementation workflow
1. Inspect only the relevant local files.
2. Make scoped edits in the child theme.
3. For new page sections, create a dynamic block under src/blocks/<slug>.
4. Put site-global UI under ai-zippy-site; put page sections under ai-zippy-page.
5. Use shared section controls for layout/background/spacing.
6. Keep all content editable and dynamic.
7. Style with Wolfgang tokens and responsive layouts.
8. Rebuild and verify.
9. Summarize changed files and checks briefly.
